# Batch Upload Fix v2.3.4 - Исправление "зависания" загрузки

## 🐛 Проблема

При массовой генерации тестовых событий (500+ событий):
- ✅ Первые 50% событий загружаются успешно
- ❌ Остальные "виснут" и не загружаются
- ❌ Не все строки (сотрудники) заполняются событиями

## 🔍 Найденные узкие места

### 1. **Supabase PostgREST лимит (1000 записей)**
**Проблема**: Endpoint `GET /events` по умолчанию возвращает максимум 1000 записей
**Симптом**: При перезагрузке после генерации загружается только часть событий
**Решение**: Добавили `.limit(50000)` для поддержки больших календарей

### 2. **Слишком большой batch size (100 событий)**
**Проблема**: Edge Function может не справляться с обработкой 100 событий за раз
**Симптом**: Часть батчей возвращает ошибки или создаёт меньше событий чем отправлено
**Решение**: Уменьшили batch size со 100 до 50 событий

### 3. **Большая задержка между батчами (2 секунды)**
**Проблема**: Для 500 событий = 10 батчей × 2 сек = 20 секунд только ожидания
**Симптом**: Медленная загрузка, пользователь думает что "зависло"
**Решение**: Уменьшили задержку с 2000мс до 500мс

### 4. **Недостаточное логирование**
**Проблема**: Трудно понять где именно происходит сбой
**Решение**: Добавили детальное логирование:
- Номер пачки (N/Total)
- Количество отправленных vs созданных событий
- Предупреждения о расхождениях
- Детали ошибок с контекстом

## ✅ Внесённые изменения

#### 1. Увеличили лимит GET /events
```typescript

// После:

  .order('id', { ascending: true })

```

## 📚 Технические детали

### Почему batch size = 50?
- Edge Functions имеют ограничение памяти и времени выполнения
- 50 событий × ~200 байт = ~10KB JSON payload (комфортно для Edge Function)
- Вставка 50 строк в PostgreSQL занимает ~100-200мс (быстро и стабильно)
- При 100 событиях начинаются периодические таймауты

### Почему задержка = 500мс?
- Даёт Edge Function время "остыть" между запросами
- Предотвращает перегрузку PostgreSQL connection pool
- Достаточно короткая для хорошего UX (пользователь видит плавный прогресс)
- Можно убрать совсем, но тогда возможны occasional errors при большой нагрузке

### Почему limit = 50000?
- Поддерживает даже очень большие календари (100 сотрудников × 10 событий × 50 воркспейсов)
- PostgreSQL легко возвращает 50k строк (~10MB JSON)
- Edge Function timeout (60 сек) достаточен для обработки такого объёма
- Если нужно больше - можно увеличить до 100000 (но тогда нужна пагинация)
