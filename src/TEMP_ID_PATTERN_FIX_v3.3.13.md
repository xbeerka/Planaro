# Исправление паттерна временных ID v3.3.13

## 🐛 Проблема

В `/contexts/SchedulerContext.tsx` строка 964 использовала **неправильный паттерн** для проверки временных ID:

```typescript
// ❌ НЕПРАВИЛЬНО - отсутствует подчеркивание в конце!
if (event.id.startsWith('e') && !event.id.startsWith('ev_temp')) {
  eventsApi.delete(event.id, accessToken).catch(err => ...)
}
```

**Проблема**: Проверка `!event.id.startsWith('ev_temp')` пропустит ID вида `ev_tempo_12345` или `ev_tempXXX`, но **НЕ пропустит** правильные временные ID вида `ev_temp_1234567890_12345`.

**Где используется правильный паттерн** (везде в коде):
- `ev_temp_${Date.now()}_${Math.floor(Math.random() * 100000)}`
- Пример: `ev_temp_1732005123456_789`

## 🔍 Последствия

**Критическая race condition**: При удалении департамента или ресурса система пыталась удалить временные события из БД через API, **ДО** того как они были созданы на сервере:

1. Пользователь создаёт событие → временный ID `ev_temp_1732005123456_789`
2. Через 5 секунд cleanup пытается удалить orphaned события
3. Проверка `!event.id.startsWith('ev_temp')` ПРОПУСКАЕТ временное событие
4. Вызов `eventsApi.delete('ev_temp_1732005123456_789', accessToken)` на сервер
5. Сервер отвечает ошибкой: `Cannot delete temporary events via API`

**Влияние**:
- ❌ Ложные ошибки в консоли (`❌ Ошибка удаления события ev_temp_XXX`)
- ❌ Нагрузка на сеть (лишние DELETE запросы)
- ⚠️ Потенциальная race condition если временное событие уже получило реальный ID на сервере

## ✅ Исправление

```typescript
// ✅ v3.3.13: КРИТИЧЕСКОЕ ИСПРАВЛЕНИЕ - правильный паттерн временных ID
if (event.id.startsWith('e') && !event.id.startsWith('ev_temp_')) {
  eventsApi.delete(event.id, accessToken).catch(err => 
    console.error(`❌ Ошибка удаления события ${event.id}:`, err)
  );
}
```

**Изменение**: Добавлено **подчеркивание в конце** паттерна: `'ev_temp'` → `'ev_temp_'`
