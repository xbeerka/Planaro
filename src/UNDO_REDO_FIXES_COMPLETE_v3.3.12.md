## Хронология исправлений

### v3.3.7: Sync history before drag ✅
**Дата**: 2025-11-15  
**Проблема**: События удалялись при Undo после быстрого drag

**Root Cause**:
- Создание события → временный ID (`ev_temp_123`)
- Быстрый drag → история сохраняет временный ID
- Сервер создаёт событие → реальный ID (`ev123`)
- Undo восстанавливает временный ID → событие не найдено → удаляется

**Решение**:
1. ✅ `flushPendingChanges()` перед drag/resize/gap drag
2. ✅ Синхронное сохранение истории через `Promise.resolve()`
3. ✅ Убран IIFE в `handlePaste` (был fire-and-forget)

**Документация**: `/SYNC_HISTORY_BEFORE_DRAG_v3.3.7.md`

---

### v3.3.8: BATCH create/update detection ✅
**Дата**: 2025-11-16  
**Проблема**: `❌ BATCH update: событие не найдено в БД`

**Root Cause**:
- ВСЕ batch операции помечались как `op: 'update'`
- Для новых событий (нет в БД) → UPDATE не работает

**Решение**:
- ✅ Определение `op: 'create' | 'update'` на основе `loadedEventIds.current.has(id)`
- ✅ Передача `id` в `data` для CREATE операций (UPSERT на сервере)
- ✅ Добавление созданных событий в `loadedEventIds`

**Документация**: `/BATCH_CREATE_UPDATE_FIX_v3.3.8.md`

---

### v3.3.9: Блокировка взаимодействий с временными событиями ✅
**Дата**: 2025-11-16  
**Проблема**: При Undo после быстрого drag временного события оно удалялось

**Root Cause**:
- Drag завершался ДО создания события на сервере
- История сохраняла временный ID

**Решение**:
- ✅ Блокировка drag/resize для событий с `id.startsWith('ev_temp_')`
- ✅ История ВСЕГДА содержит реальные ID
- ✅ Задержка ~500ms между созданием и drag

**Документация**: `/TEMP_EVENTS_INTERACTION_BLOCK_v3.3.9.md`

---

### v3.3.10: Конфликт Undo и Debounced Save ✅
**Дата**: 2025-11-17  
**Проблема**: `❌ BATCH update: событие e37367 не найдено в БД` при Undo

**Root Cause**:
- Создание события → добавляется в debounced save queue (500ms)
- Undo → событие удаляется с сервера
- Debounced save срабатывает → пытается UPDATE удалённого события

**Решение**:
- ✅ Очистка pending операций для удалённых событий в `handleUndo()` и `handleRedo()`
- ✅ Вызов `cancelPendingChange()` для каждого удалённого события
- ✅ Добавлена зависимость `cancelPendingChange` в useCallback

**Документация**: `/UNDO_DEBOUNCED_SAVE_CONFLICT_FIX_v3.3.10.md`

---

### v3.3.11: Race Condition в Undo/Redo ✅
**Дата**: 2025-11-17  
**Проблема**: Race condition при быстром нажатии Ctrl+Z несколько раз подряд

**Root Cause**:
- Второй Undo запускался ДО завершения первого
- Конфликты синхронизации с сервером

**Решение**:
- ✅ Добавлен `isUndoRedoInProgressRef` для блокировки одновременных операций
- ✅ Проверка блокировки в начале `handleUndo` и `handleRedo`
- ✅ Гарантированное снятие блокировки через `finally` блок
- ✅ Логирование блокировки для диагностики

**Документация**: `/UNDO_REDO_RACE_CONDITION_FIX_v3.3.11.md`

---

### v3.3.12: Блокировка Undo для pending событий ✅ ТЕКУЩАЯ ВЕРСИЯ
**Дата**: 2025-11-18  
**Проблема**: События "воскресали" после быстрого Undo сразу после создания

**Root Cause**:
```
T+0ms:   createEvent() → запрос на сервер
T+200ms: Ctrl+Z → событие удалено из стейта
T+500ms: createEvent завершён → событие в БД
T+4000ms: Delta Sync → событие "воскресло"
```

**Решение**:
- ✅ Проверка наличия pending событий (`id.startsWith('ev_temp_')`) в начале `handleUndo` и `handleRedo`
- ✅ Блокировка Undo/Redo если есть события в процессе создания
- ✅ Toast уведомление "Подождите, дождитесь завершения создания событий"
- ✅ Типичная задержка ~500ms

**Документация**: 
- `/UNDO_PENDING_EVENTS_FIX_v3.3.12.md`
- `/QUICK_TEST_UNDO_PENDING_v3.3.12.md`
- `/RELEASE_NOTES_v3.3.12.md`
