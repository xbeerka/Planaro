# Исправление множественного восстановления событий через Undo/Redo

## 🐛 Проблема

При быстром удалении нескольких событий с последующим восстановлением через Redo (или при восстановлении десятка событий подряд) возникали race conditions и нестабильное поведение.

### Симптомы
- События "мигают" при восстановлении
- Некоторые события могут не восстановиться
- Дубликаты событий
- Ошибки "событие не найдено в БД" в консоли
- Нестабильное обновление ID в истории

### Причины
1. **Параллельное создание**: Все события восстанавливались одновременно через `Promise.all()`
2. **Race condition в state**: Каждый `createEvent` независимо обновлял `setEventsState`, что могло привести к потере обновлений
3. **Конфликт временных ID**: `Date.now()` мог генерировать одинаковые ID для событий созданных в одну миллисекунду
4. **Конфликт с Delta Sync**: `lastLocalChangeRef` обновлялся после каждого события, но Delta Sync мог запуститься между операциями
5. **Async updateHistoryEventId**: Обновление ID в истории происходило асинхронно в `.then()`, порядок не гарантирован

## ✅ Решение

### 1. Последовательное восстановление событий

**БЫЛО** (параллельное):
```typescript
eventsToRestore.forEach(event => {
  promises.push(
    createEvent(eventWithoutId).then(newEvent => {
      updateHistoryEventId(oldId, newEvent.id);
    })
  );
});
```

**СТАЛО** (последовательное):
```typescript
if (eventsToRestore.length > 0) {
  console.log(`🔄 Batch restore: восстанавливаем ${eventsToRestore.length} событий ПОСЛЕДОВАТЕЛЬНО...`);
  
  const batchRestorePromise = (async () => {
    for (const event of eventsToRestore) {
      const oldId = event.id;
      console.log(`🔄 Восстановление события ${oldId} (${eventsToRestore.indexOf(event) + 1}/${eventsToRestore.length})...`);
      
      try {
        const { id, ...eventWithoutId } = event;
        const newEvent = await createEvent(eventWithoutId);
        
        console.log(`✅ Событие восстановлено: ${oldId} → ${newEvent.id}`);
        
        // Обновляем ID во ВСЕЙ истории
        updateHistoryEventId(oldId, newEvent.id);
      } catch (err) {
        console.error(`❌ Ошибка восстановления события ${oldId}:`, err);
      }
    }
    console.log(`✅ Batch restore завершён: ${eventsToRestore.length} событий`);
  })();
  
  promises.push(batchRestorePromise);
}
```

### 2. Уникальная генерация временных ID

**БЫЛО**:
```typescript
id: event.id || `ev_temp_${Date.now()}`
```

**СТАЛО**:
```typescript
// ✅ Date.now() + Math.random() предотвращает коллизии
id: event.id || `ev_temp_${Date.now()}_${Math.random().toString(36).substring(2, 9)}`
```

### 3. Улучшенное логирование

Добавлены детальные логи для отладки:
- `🔄 Batch restore: восстанавливаем N событий ПОСЛЕДОВАТЕЛЬНО...`
- `🔄 Восстановление события {oldId} ({X}/{N})...`
- `✅ Событие восстановлено: {oldId} → {newId}`
- `✅ Batch restore завершён: N событий`

## 📊 Производительность

### Влияние на скорость
- **Параллельное** (было): ~1-2 секунды для 10 событий
- **Последовательное** (стало): ~2-3 секунды для 10 событий

**Компромисс**: Немного медленнее, но СТАБИЛЬНО и БЕЗ багов.

### Оптимизация
- Удаления и обновления выполняются ПАРАЛЛЕЛЬНО (быстро)
- Только восстановления выполняются ПОСЛЕДОВАТЕЛЬНО (стабильно)
- Delta Sync защищён через `lastLocalChangeRef`

## 🎓 Выводы

### Что мы узнали
1. **Promise.all() опасен для state updates**: Параллельные `setState` могут конфликтовать
2. **Date.now() не уникален**: Нужна рандомизация для ID генерируемых в один момент
3. **Последовательность важнее скорости**: Для критичных операций стабильность > производительность
4. **Логирование спасает**: Детальные логи помогают быстро найти проблему

### Best Practices
- ✅ Используй `for await` для последовательных async операций с state
- ✅ Добавляй рандомизацию к timestamp для уникальных ID
- ✅ Логируй progress для batch операций
- ✅ Защищай критичные операции от race conditions
- ✅ Тестируй edge cases (быстрое нажатие, много операций)
