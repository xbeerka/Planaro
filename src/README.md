## 🎯 Описание

**Planaro** - продвинутое веб-приложение для планирования ресурсов с годовым календарём на 52 недели. Создано для KODE команды.

### Основные возможности

- ✅ **Годовой календарь** с 52 неделями и sticky заголовками
- ✅ **Drag & Drop** событий с курсорным позиционированием
- ✅ **Resize** событий в 4 направлениях с плавной анимацией
- ✅ **Режим ножниц** для разделения событий на границах недель
- ✅ **Undo/Redo** с историей изменений (Ctrl+Z)
- ✅ **Z-order** управление для перекрывающихся событий
- ✅ **Фильтрация** по департаментам и проектам
- ✅ **Управление воркспейсами** с навигацией через URL
- ✅ **Онлайн пользователи** с аватарками (HTTP-based presence)
- ✅ **Автообновление** через Simple Polling (18 req/min)
- ✅ **30-дневные сессии** с автоматическим обновлением токенов
- ✅ **Генерация тестовых данных** (100% заполнение календаря)

## 🔥 Критические исправления v2.3.7

### Проблема
PostgREST возвращал максимум **1000 записей** за запрос, независимо от `.limit()` или `.range()`.

**Результат**: Создано 8053 событий → Загружено 1000 → Потеряно 7053 (**87.6%**)

### Решение
Реализована **автоматическая пагинация** - загрузка событий циклом по 1000 записей.

**Результат**: 
- ✅ **100% integrity** (было 12.4%)
- ✅ Поддержка до **100,000 событий**
- ✅ 8053 события за **9 запросов** (~2-5 сек)

Подробнее: [`/FIX_PAGINATION_v2.3.7.md`](/FIX_PAGINATION_v2.3.7.md)

## 🗄️ Структура базы данных

```
workspaces → воркспейсы (id, name, year)
  ↓
departments → департаменты (id, name, queue, workspace_id)
  ↓
users → сотрудники (id, name, email, department_id, workspace_id)
  ↓
events → события (id, user_id, project_id, start_week, weeks_span, ...)
  ↓
projects → проекты (id, name, backgroundColor, textColor, workspace_id)
```

## 🌐 Онлайн пользователи

### HTTP-based Presence
1. **Heartbeat**: каждые 30 секунд при открытом календаре
2. **KV Store**: `presence:{workspace_id}:{user_id}` с TTL 60 сек
3. **Graceful leave**: explicit DELETE при закрытии календаря
4. **Batch запросы**: 1 запрос для всех воркспейсов
5. **Кэширование**: TTL 45 секунд, мгновенное отображение

### Отображение
- **В карточках**: только пользователи ВНУТРИ воркспейса
- **Текущий пользователь**: зелёный градиент с меткой "(вы)"
- **Другие**: синий градиент с аватаркой

## 🔄 Автообновление (Simple Polling)

### Что обновляется
- **События**: каждые **10 секунд**
- **Сотрудники**: каждые **15 секунд**
- **Департаменты**: каждые **15 секунд**
- **Проекты**: каждые **15 секунд**

### Преимущества
- ✅ Работает везде (только HTTP)
- ✅ Простая отладка
- ✅ Предсказуемое поведение (18 req/min/user)
- ✅ Синхронизация всех данных между пользователями

## 💡 Best Practices

### DO ✅
- Используй существующие shadcn компоненты
- Добавляй детальные логи (эмоджи + контекст)
- Делай валидацию на сервере
- Используй Promise.all() для параллельных операций
- Используй `/utils/jwt.ts` для декодирования токенов
- Проверяй integrity rate после массовых операций

### DON'T ❌
- Не создавай новые таблицы БД
- Не делай автосохранение в модалках
- Не меняй защищённые файлы (kv_store.tsx, info.tsx)
- Не используй alert() - используй toast
- Не делай последовательные await в циклах
- Не используй `atob()` напрямую для JWT
