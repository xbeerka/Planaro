# Тестирование мгновенной очистки кэша при возврате из календаря

## Проблема до оптимизации

**Симптом**: "Мигание" текущего пользователя при возврате из календаря.

### Scenario без оптимизации:

```
1. Пользователь test@kode.ru открывает календарь воркспейса #14
   ↓
2. OnlineUsers отправляет heartbeat → presence сохраняется на сервере
   ↓
3. WorkspaceListScreen (в другой вкладке) делает batch запрос
   ↓ 
4. Кэш cache_online_users_batch содержит: { "14": [{ email: "test@kode.ru", ... }] }
   ↓
5. Пользователь нажимает "Назад" → возврат к WorkspaceListScreen
   ↓
6. OnlineUsers размонтируется → sendLeave() удаляет presence на сервере
   ↓
7. WorkspaceListScreen монтируется → читает СТАРЫЙ кэш
   ↓
8. ❌ test@kode.ru видит свою аватарку 1-2 секунды (данные из кэша)
   ↓
9. Batch запрос обновляется → кэш обновлён → аватарка исчезает
   ↓
10. ❌ РЕЗУЛЬТАТ: "Мигание" аватарки (появилась → исчезла)
```

## Решение с оптимизацией (v2)

### Scenario с оптимизацией v2 (защита от быстрых переходов):

```
1. Пользователь test@kode.ru открывает календарь воркспейса #14
   ↓
2. OnlineUsers отправляет heartbeat → presence сохраняется на сервере
   ↓
3. WorkspaceListScreen (в другой вкладке) делает batch запрос
   ↓
4. Кэш cache_online_users_batch содержит: { "14": [{ email: "test@kode.ru", ... }] }
   ↓
5. Пользователь нажимает "Назад" → вызывается handleBackToWorkspaces()
   ↓
6. ✅ handleBackToWorkspaces() читает кэш → удаляет test@kode.ru → сохраняет
   ↓
7. ✅ handleBackToWorkspaces() устанавливает блокировку suppress_current_user_presence (TTL 5 сек)
   ↓
8. Обновлённый кэш: { "14": [] }
   ↓
9. OnlineUsers размонтируется → sendLeave() удаляет presence на сервере
   ↓
10. WorkspaceListScreen монтируется → читает кэш
   ↓
11. ✅ Кэш проверяет блокировку → фильтрует test@kode.ru (даже если он там есть)
   ↓
12. ✅ test@kode.ru НЕ видит свою аватарку
   ↓
13. Batch запрос выполняется (может быть ДО того как leave дошёл)
   ↓
14. ✅ Batch данные проверяют блокировку → фильтруют test@kode.ru
   ↓
15. ✅ Даже если presence ещё на сервере → он НЕ отображается (блокировка активна)
   ↓
16. Leave доходит → presence удаляется на сервере
   ↓
17. Через 5 сек блокировка истекает (уже не нужна - presence удалён)
   ↓
18. ✅ РЕЗУЛЬТАТ: Никакого "мигания" даже при мгновенных переходах
```

## Ключевые моменты (v2)

### Двухуровневая защита

**Уровень 1 - Очистка кэша** (работает для обычных случаев):
- Удаляет текущего пользователя из локального кэша
- Предотвращает показ устаревших данных при монтировании

**Уровень 2 - Временная блокировка** (защита от "гонки условий"):
- Фильтрует текущего пользователя даже если он пришёл с сервера
- Решает проблему: быстро вошли → heartbeat → быстро вышли → batch запрос успел выполниться
- Без блокировки: кэш очищен, но batch записал свежие данные с сервера (где presence ещё есть)
- С блокировкой: batch данные фильтруются → текущий пользователь не показывается

### Timing критичен

**Порядок операций в `handleBackToWorkspaces()` (v2)**:

```typescript
const handleBackToWorkspaces = async () => {
  console.log('🔙 Возврат к списку воркспейсов');
  
  // 1. ПЕРВЫМ ДЕЛОМ - двухуровневая защита
  try {
    const currentUserEmail = getEmailFromToken(accessToken);
    const cachedData = await getStorageJSON('cache_online_users_batch');
    
    if (cachedData?.data) {
      // 1.1. Очистка кэша (уровень 1)
      const updatedData = { ...cachedData.data };
      for (const workspaceId in updatedData) {
        updatedData[workspaceId] = updatedData[workspaceId].filter(
          user => user.email !== currentUserEmail
        );
      }
      
      await setStorageJSON('cache_online_users_batch', {
        data: updatedData,
        timestamp: cachedData.timestamp
      });
    }
    
    // 1.2. Установка временной блокировки (уровень 2)
    // Защита от "гонки условий" при быстрых переходах
    await setStorageJSON('suppress_current_user_presence', {
      email: currentUserEmail,
      timestamp: Date.now(),
      ttl: 5000 // 5 секунд
    });
    console.log('🔒 Блокировка установлена - текущий пользователь будет скрыт 5 сек');
  } catch (error) {
    console.warn('⚠️ Ошибка защиты от мигания');
  }
  
  // 2. ЗАТЕМ - обычная логика возврата
  setSelectedWorkspace(null);
  window.history.pushState(null, '', '/');
  document.title = 'Planaro - Управление рабочими пространствами';
};
```

**Почему важно очищать ДО `setSelectedWorkspace(null)`**:
- `setSelectedWorkspace(null)` → триггерит ре-рендер → размонтирует SchedulerMain → монтирует WorkspaceListScreen
- Если очистить ПОСЛЕ → WorkspaceListScreen смонтируется с СТАРЫМ кэшем
- Если очистить ДО → WorkspaceListScreen смонтируется с НОВЫМ (очищенным) кэшем

### IndexedDB асинхронность

**Важно**: `getStorageJSON` и `setStorageJSON` - асинхронные операции.

```typescript
// ❌ НЕПРАВИЛЬНО - очистка может не успеть
const handleBackToWorkspaces = () => {
  // Очистка кэша без await
  cleanCache(); // Promise не дождались
  
  setSelectedWorkspace(null); // ← Ре-рендер может произойти ДО очистки
};

// ✅ ПРАВИЛЬНО - ждём очистки
const handleBackToWorkspaces = async () => {
  await cleanCache(); // ← Дождались завершения
  
  setSelectedWorkspace(null); // ← Ре-рендер происходит ПОСЛЕ очистки
};
```

## Тестовые сценарии

### Сценарий 1: Быстрый переход (КРИТИЧНЫЙ - проверка блокировки)

**Шаги**:
1. Откройте приложение, войдите как `test@kode.ru`
2. **Быстро** откройте календарь воркспейса #14 (клик)
3. **Мгновенно** нажмите "Назад" (<1 секунды после входа)
4. **Наблюдайте**: НЕ должно быть "мигания" вашей аватарки

**Что происходит**:
- За <1 сек OnlineUsers успевает отправить heartbeat → presence сохраняется на сервере
- handleBackToWorkspaces() очищает кэш + устанавливает блокировку
- Batch запрос может выполниться ДО того как leave дошёл
- **БЕЗ блокировки**: batch записал бы presence в кэш → "мигание"
- **С блокировкой**: batch данные фильтруются → нет "мигания"

**Ожидаемые логи**:
```
🔙 Возврат к списку воркспейсов
🧹 Очистка текущего пользователя из presence кэша: test@kode.ru
✅ Кэш очищен от текущего пользователя
🔒 Установлена временная блокировка presence для: test@kode.ru
[... batch запрос выполняется ...]
🔒 Активна блокировка presence для test@kode.ru (0с / 5с)
  🚫 Workspace 14: отфильтрован текущий пользователь (1 удалено)
✅ Batch данные отфильтрованы - текущий пользователь скрыт
```

**Проверка**:
- ✅ Аватарка НЕ появляется даже на долю секунды
- ✅ Логи показывают активацию блокировки
- ✅ Плавный переход без визуальных артефактов

## Отладка проблем

### Проблема: "Мигание" всё равно есть

**Возможные причины**:

1. **Очистка кэша происходит ПОСЛЕ setSelectedWorkspace**:
   - Проверьте порядок вызовов в `handleBackToWorkspaces`
   - Очистка должна быть ДО `setSelectedWorkspace(null)`

2. **Нет await на очистку кэша**:
   - Убедитесь что функция `handleBackToWorkspaces` async
   - IndexedDB операции должны ждать завершения

3. **Ошибка в try-catch блоке**:
   - Проверьте логи: есть ли `⚠️ Ошибка очистки кэша`
   - Если есть - смотрите детали ошибки

**Отладка**:
```typescript
const handleBackToWorkspaces = async () => {
  console.log('🔙 НАЧАЛО возврата');
  
  try {
    console.log('1️⃣ Начало очистки кэша');
    const before = await getStorageJSON('cache_online_users_batch');
    console.log('   Кэш ДО:', before);
    
    // ... очистка ...
    
    const after = await getStorageJSON('cache_online_users_batch');
    console.log('   Кэш ПОСЛЕ:', after);
    console.log('2️⃣ Очистка завершена');
  } catch (error) {
    console.error('❌ ОШИБКА очистки:', error);
  }
  
  console.log('3️⃣ Вызов setSelectedWorkspace(null)');
  setSelectedWorkspace(null);
  
  console.log('🔙 КОНЕЦ возврата');
};
```

### Проблема: Другие пользователи пропадают

**Симптом**: При возврате из календаря исчезают ВСЕ пользователи, а не только текущий.

**Причина**: Неправильный фильтр в очистке кэша.

**Проверка**:
```typescript
// ❌ НЕПРАВИЛЬНО - удаляет всех
updatedData[workspaceId] = [];

// ✅ ПРАВИЛЬНО - удаляет только текущего
updatedData[workspaceId] = users.filter(
  user => user.email !== currentUserEmail
);
```

### Проблема: Кэш не очищается совсем

**Симптом**: Логи показывают очистку, но "мигание" есть.

**Возможные причины**:

1. **WorkspaceListScreen читает кэш ДО очистки**:
   - Проверьте timing - возможно ре-рендер триггерится раньше
   - Убедитесь что очистка завершается до `setSelectedWorkspace`

2. **Кэш перезаписывается batch запросом**:
   - Batch запрос может вернуть старые данные если сервер не обновился
   - Проверьте что `sendLeave()` успел дойти до сервера

3. **IndexedDB транзакция не завершилась**:
   - IndexedDB может буферизовать запись
   - Добавьте задержку для теста: `await new Promise(r => setTimeout(r, 100))`

## Performance метрики

### Замеры времени

**До оптимизации**:
- Возврат из календаря → список: ~200ms (ре-рендер)
- "Мигание" аватарки: 1000-2000ms (до batch обновления)
- **UX оценка**: ⭐⭐⭐ (заметное "мигание")

**После оптимизации**:
- Возврат из календаря → список: ~250ms (ре-рендер + очистка кэша)
- "Мигание" аватарки: 0ms (нет мигания)
- **UX оценка**: ⭐⭐⭐⭐⭐ (плавный переход)

**Overhead очистки кэша**: +50ms
**Выигрыш UX**: устранение 1-2 секунд визуального артефакта

### Memory footprint

**IndexedDB операции**:
- Чтение кэша: ~5-10ms
- Обновление кэша: ~10-20ms
- Итого: ~15-30ms overhead

**Альтернатива без оптимизации**:
- Ждать batch запрос: 0-15000ms (зависит от таймера)
- Визуальный артефакт: заметен пользователю

**Вывод**: 30ms overhead за устранение визуального артефакта - отличный trade-off.
