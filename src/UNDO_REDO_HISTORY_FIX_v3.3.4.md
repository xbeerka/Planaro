# Исправление проблемы с историей Undo/Redo v3.3.4

## 🐛 Проблема

При быстром удалении нескольких событий (5 событий за 5 секунд) и последующих операциях Undo/Redo история содержала только **2 записи** вместо ожидаемых 10+ записей. Это приводило к невозможности полного отката изменений.

### Сценарий воспроизведения:
1. Имеем 9 событий (8 в ряд + 1 отдельное)
2. Быстро удаляем 5 событий (по 1 в секунду)
3. Делаем 3 Undo (восстанавливаем 3 события)
4. Изменяем высоту одного события
5. Ждём 10 секунд
6. Быстро нажимаем Undo до конца → **видим только 2 записи в истории**
7. Не можем вернуться к начальному состоянию

### Логи показали:
```
История: текущий index 1, всего записей 2  ← ПРОБЛЕМА!
📥 Delta Sync: получено 1 изменений        ← Delta Sync вмешивается
✅ Delta Sync: применено 1 изменений        ← Перезаписывает локальные изменения
```

## 🔍 Причины

### Причина 1: Асинхронное сохранение истории при удалении
В `handleContextDelete` функция `saveHistory` вызывалась **после** асинхронного `deleteEvent`:

```typescript
// ❌ БЫЛО (строка 924):
(async () => {
  try {
    await deleteEvent(eventId);           // Удаление на сервере
    saveHistory(newEvents, newEventZOrder, projects); // ← Только после успешного удаления
  } catch (error) {
    // откат
  }
})();
```

**Проблема**: При быстром удалении 5 событий:
- Удаление 1: `deleteEvent` → `saveHistory` (асинхронно)
- Удаление 2: `deleteEvent` → `saveHistory` (асинхронно) ← **перезаписывает предыдущее**
- Удаление 3: `deleteEvent` → `saveHistory` (асинхронно) ← **перезаписывает предыдущее**
- Удаление 4: `deleteEvent` → `saveHistory` (асинхронно) ← **перезаписывает предыдущее**
- Удаление 5: `deleteEvent` → `saveHistory` (асинхронно) ← **перезаписывает предыдущее**

**Результат**: История содержит только **последнее** удаление, все промежуточные состояния потеряны.

### Причина 2: Недостаточная блокировка Delta Sync
Блокировка Delta Sync составляла **2 секунды**, но пользователь делал быстрые Undo/Redo (быстрее чем за 2 секунды):

```typescript
// ❌ БЫЛО:
if (timeSinceLastChange < 2000) {  // 2 секунды
  console.log('⏸️ Delta Sync: пропуск (недавнее локальное изменение)');
  return;
}
```

**Проблема**: 
- Пользователь делает Undo/Redo каждые 0.5-1 секунду
- Блокировка истекает через 2 секунды
- Delta Sync загружает данные с сервера и перезаписывает локальные изменения
- Следующий Undo/Redo восстанавливает неправильное состояние

## ✅ Решение

### Исправление 1: Синхронное сохранение истории ПЕРЕД удалением

```typescript
// ✅ СТАЛО (строка 920-937):
// ✅ КРИТИЧНО: Сохраняем историю ПЕРЕД удалением на сервере (синхронно!)
// Это гарантирует что при быстром удалении нескольких событий
// все промежуточные состояния сохранятся в истории
saveHistory(newEvents, newEventZOrder, projects);
console.log('📝 История сохранена перед удалением события:', eventId);

// Для реальных событий: deleteEvent делает оптимистичное удаление из state
(async () => {
  try {
    await deleteEvent(eventId);
    console.log('✅ Событие успешно удалено:', eventId);
  } catch (error) {
    console.error("❌ Ошибка удаления события из БД:", error);
    // deleteEvent уже откатил изменения в events, восстанавливаем z-order
    if (zOrderToRestore !== undefined) {
      setEventZOrder((prev) =>
        new Map(prev).set(eventId, zOrderToRestore),
      );
    }
    // ❌ ОТКАТЫВАЕМ историю (делаем Undo при ошибке)
    historyUndo();
    console.log('↩️ История откачена из-за ошибки удаления');
  }
})();
```

**Преимущества**:
- ✅ История сохраняется **мгновенно** при каждом удалении
- ✅ При быстром удалении 5 событий → **5 записей** в истории
- ✅ При ошибке удаления автоматический откат через `historyUndo()`
- ✅ Полная трассировка всех промежуточных состояний

## 🎯 Рекомендации

1. **Мониторинг**: Проверить логи в продакшене на наличие:
   - Частых `historyUndo()` из-за ошибок удаления (может указывать на проблемы с сетью)
