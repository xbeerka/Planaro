## 🐛 Исправленная проблема

### Проблема: События "воскресают" после быстрого Undo

**Описание бага:**
При создании события и быстром нажатии Ctrl+Z (до завершения создания на сервере) событие исчезало с календаря, но через несколько секунд "воскресало" обратно.

**Root Cause:**
Race condition между тремя операциями:
1. `createEvent()` → отправка запроса на сервер (~500ms)
2. `Undo` → удаление события из локального стейта
3. `Delta Sync` → загрузка новых событий из БД каждые 4 секунды

**Результат:**
- Пользователь делает Undo → событие удаляется локально
- Сервер завершает создание → событие появляется в БД
- Delta Sync загружает событие из БД → событие "воскресает"

---

## ✅ Решение

### Блокировка Undo/Redo для pending событий

**Изменения:**
- Добавлена проверка наличия событий с временными ID (`ev_temp_*`) в начало `handleUndo()` и `handleRedo()`
- При обнаружении pending событий показывается toast уведомление
- Undo/Redo блокируется до завершения всех pending операций (~500ms)

**Код:**
```typescript
const handleUndo = useCallback(async () => {
  // ✅ v3.3.12: Блокируем Undo если есть pending события
  const hasPendingEvents = events.some(e => e.id.startsWith('ev_temp_'));
  if (hasPendingEvents) {
    showToast({
      title: 'Подождите',
      description: 'Дождитесь завершения создания событий',
      variant: 'warning'
    });
    return;
  }
  // ... обычная логика Undo
}, [events, showToast, ...]);
```

## 📝 Известные ограничения

### Задержка ~500ms
- **Что**: Пользователь должен подождать завершения создания перед Undo
- **Почему**: Нельзя отменить уже запущенный запрос на сервере
- **Приемлемо**: Да, задержка минимальная и с понятным сообщением

### Множественные pending события
- **Что**: Если создано несколько событий подряд, Undo блокируется до завершения ВСЕХ
- **Почему**: Нельзя частично делать Undo (часть событий pending, часть готовы)
- **Приемлемо**: Да, редкий сценарий
