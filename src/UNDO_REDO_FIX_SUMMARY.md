# 🔧 Исправление: Защита истории от сохранения событий без проектов

## 📌 Краткое описание

**Проблема**: При использовании Undo после сохранения изменений в модалках Users/Projects система выдавала ошибку о попытке восстановить состояние с событиями но без проектов.

**Решение**: Исправлены все вызовы `resetHistory()` для передачи проектов + добавлена дополнительная защита в `saveHistory()`.

**Статус**: ✅ Исправлено и задокументировано

---

## 🔍 Детальный анализ проблемы

### Что происходило?

1. **Триггер**: Пользователь открывал модалку Users или Projects, вносил изменения и сохранял
2. **Колбек onResetHistory**: Вызывался `resetHistory(events, eventZOrder)` БЕЗ третьего параметра `projects`
3. **Сохранение в историю**: История сохранялась с событиями но с пустым массивом проектов
4. **Попытка Undo**: Пользователь нажимал Ctrl+Z
5. **Защита сработала**: Система правильно блокировала восстановление некорректного состояния
6. **Ошибка в консоли**:
   ```
   ❌ История: КРИТИЧЕСКАЯ ОШИБКА - попытка восстановить state с событиями но без проектов!
   ❌ История: это приведёт к удалению всех событий. Отменяем Undo.
   ```

### Почему это проблема?

- **Пользовательский опыт**: Ошибка в консоли выглядит пугающе
- **Функциональность**: Undo не работал после сохранения в модалках
- **Потенциальная потеря данных**: Если бы защита не сработала, все события были бы удалены

### Корневая причина

**Неправильные вызовы resetHistory** в SchedulerMain.tsx:

```typescript
// ❌ НЕПРАВИЛЬНО (было)
onResetHistory={() => {
  resetHistory(events, eventZOrder); // Проекты не передаются!
}}
```

**Fallback логика в saveHistory** не спасала:
```typescript
// В useHistory.ts
if (projects.length === 0) {
  const prevProjects = history[historyIndex]?.projects || [];
  if (prevProjects.length > 0) {
    projectsToSave = prevProjects; // Берём из предыдущего состояния
  } else {
    projectsToSave = []; // ❌ Сохраняем пустой массив!
  }
}
```

Если предыдущее состояние тоже было без проектов (начальная загрузка), fallback не помогал.

---

## ✅ Решение

### 1. Исправлены вызовы resetHistory

**Файл**: `/components/scheduler/SchedulerMain.tsx`

**UsersManagementModal**:
```typescript
// ✅ ПРАВИЛЬНО (после исправления)
onResetHistory={() => {
  console.log('📝 Сброс истории после изменений сотрудников (с проектами)');
  resetHistory(events, eventZOrder, projects); // Передаём проекты!
}}
```

**ProjectsManagementModal**:
```typescript
// ✅ ПРАВИЛЬНО (после исправления)
onResetHistory={() => {
  console.log('📝 Сброс истории после изменений проектов (с проектами)');
  resetHistory(events, eventZOrder, projects); // Передаём проекты!
}}
```

### 2. Усилена защита в saveHistory

**Файл**: `/hooks/useHistory.ts`

**Добавлена проверка**:
```typescript
if (projects.length === 0) {
  const prevProjects = history[historyIndex]?.projects || [];
  
  if (prevProjects.length > 0) {
    projectsToSave = prevProjects;
  } else {
    // ✅ КРИТИЧЕСКАЯ ПРОВЕРКА
    if (events.length > 0) {
      console.error('❌ История: КРИТИЧЕСКАЯ ОШИБКА - попытка сохранить события без проектов!');
      console.error('❌ История: это приведёт к некорректному Undo/Redo. Отменяем сохранение.');
      return; // ⛔ НЕ сохраняем некорректное состояние!
    }
    
    // Если нет ни событий ни проектов - это начальная загрузка, это OK
    projectsToSave = [];
  }
}
```

### 3. Улучшено логирование

**Детальные логи при сбросе истории**:
```typescript
console.log('📝 Сброс истории после изменений сотрудников (с проектами)');
console.log('📝 Сброс истории после изменений проектов (с проектами)');
```

Теперь сразу видно что проекты передаются правильно.
