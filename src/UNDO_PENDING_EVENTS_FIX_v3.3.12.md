# Исправление: Блокировка Undo/Redo для pending событий (v3.3.12)

## 🎯 Проблема

**Race condition между Undo и pending CREATE операцией:**

### Сценарий воспроизведения
1. Пользователь создаёт событие (двойной клик, вставка, перетаскивание)
2. Событие получает временный ID (`ev_temp_123...`) и показывается со спиннером
3. Запрос `createEvent` отправлен на сервер, но ещё не завершился (~500ms)
4. Пользователь быстро делает Ctrl+Z (Undo)
5. **Проблема**: Событие исчезает из локального стейта, но запрос продолжает работать
6. Сервер создаёт событие в БД (возвращает реальный ID)
7. Delta Sync через 4 секунды загружает новое событие из БД
8. **Результат**: Событие "воскресает" на календаре вопреки Undo

### Почему это происходит
```typescript
// Временная шкала race condition:

T+0ms:   createEvent(tempEvent) отправлен на сервер
         ↓
         Событие в локальном стейте: { id: 'ev_temp_123...', ... }
         Отображается со спиннером (pending состояние)
         ↓
T+200ms: Пользователь нажимает Ctrl+Z
         ↓
         handleUndo() восстанавливает предыдущее состояние БЕЗ этого события
         setEvents(previousEvents) // событие удалено из стейта
         ↓
T+500ms: createEvent ЗАВЕРШАЕТСЯ на сервере ✅
         ↓
         Событие создано в БД с реальным ID: { id: 'ev123', ... }
         НО локальный стейт уже НЕ содержит это событие (удалено Undo)
         ↓
T+4000ms: Delta Sync запрашивает изменения с сервера
         ↓
         GET /events/changes?since=... → { events: [{ id: 'ev123', ... }] }
         ↓
         Событие добавляется обратно в локальный стейт
         ↓
         🐛 BUG: Событие "воскресло" вопреки Undo!
```

### Варианты решения (рассмотренные)

#### ❌ Вариант 1: Отменять pending CREATE запросы
```typescript
// Проблема: Нельзя отменить уже запущенный fetch() без AbortController
const handleUndo = async () => {
  pendingEvents.forEach(event => {
    // Как отменить запрос который уже на сервере?
    // AbortController отменит запрос на клиенте, но НЕ на сервере
    // Сервер всё равно создаст событие в БД
  });
};
```

#### ❌ Вариант 2: Помечать события для удаления после создания
```typescript
// Проблема: Сложная логика с дополнительным Map для "отложенного удаления"
const pendingDeletions = new Map<string, boolean>();

const createEvent = async (event) => {
  const realId = await api.create(event);
  
  // Проверить флаг после создания
  if (pendingDeletions.has(tempId)) {
    await api.delete(realId); // сразу удалить
    pendingDeletions.delete(tempId);
  }
};
```

#### ✅ Вариант 3: Блокировать Undo/Redo пока есть pending события (ВЫБРАНО)
```typescript
// Простое и безопасное решение
const handleUndo = async () => {
  // Проверяем есть ли события с временными ID
  const hasPendingEvents = events.some(e => e.id.startsWith('ev_temp_'));
  
  if (hasPendingEvents) {
    // Показываем понятное сообщение пользователю
    showToast({ 
      title: 'Подождите',
      description: 'Дождитесь завершения создания событий',
      variant: 'warning'
    });
    return; // блокируем Undo
  }
  
  // ... обычная логика Undo
};
```

**Почему Вариант 3 лучший:**
1. ✅ **Простота** - 5 строк кода без сложной логики
2. ✅ **Безопасность** - гарантированно нет race conditions
3. ✅ **UX** - понятное сообщение объясняет причину блокировки
4. ✅ **Задержка** - обычно ~500ms, что приемлемо для пользователя
5. ✅ **Надёжность** - работает для любых pending операций (paste, drag, double-click)

## ✅ Решение

### Добавлена проверка pending событий в начало `handleUndo()` и `handleRedo()`

#### Файл: `/components/scheduler/SchedulerMain.tsx`

```typescript
const handleUndo = useCallback(async () => {
  // ✅ v3.3.12: КРИТИЧНО - блокируем Undo если есть pending события
  const hasPendingEvents = events.some(e => e.id.startsWith('ev_temp_'));
  if (hasPendingEvents) {
    console.warn('⏸️ UNDO: Заблокировано - есть события в процессе создания');
    showToast({
      title: 'Подождите',
      description: 'Дождитесь завершения создания событий',
      variant: 'warning'
    });
    return;
  }
  
  // ✅ v3.3.11: КРИТИЧНО - блокируем одновременные Undo операции
  if (isUndoRedoInProgressRef.current) {
    console.warn('⏸️ UNDO/REDO: Undo уже выполняется, ожидайте завершения...');
    return;
  }
  
  // ... остальная логика Undo
}, [events, showToast, /* ... */]);

const handleRedo = useCallback(async () => {
  // ✅ v3.3.12: КРИТИЧНО - блокируем Redo если есть pending события
  const hasPendingEvents = events.some(e => e.id.startsWith('ev_temp_'));
  if (hasPendingEvents) {
    console.warn('⏸️ REDO: Заблокировано - есть события в процессе создания');
    showToast({
      title: 'Подождите',
      description: 'Дождитесь завершения создания событий',
      variant: 'warning'
    });
    return;
  }
  
  // ✅ v3.3.11: КРИТИЧНО - блокируем одновременные Redo операции
  if (isUndoRedoInProgressRef.current) {
    console.warn('⏸️ UNDO/REDO: Redo уже выполняется, ожидайте завершения...');
    return;
  }
  
  // ... остальная логика Redo
}, [events, showToast, /* ... */]);
```

### Логика работы

1. **При нажатии Ctrl+Z (Undo)**:
   - Проверяется наличие событий с `id.startsWith('ev_temp_')`
   - Если есть → блокируем Undo + показываем toast с предупреждением
   - Если нет → выполняем обычный Undo

2. **При нажатии Ctrl+Shift+Z (Redo)**:
   - Аналогичная проверка на pending события
   - Блокируем Redo если есть pending операции

3. **Типичная задержка**:
   - Создание события: ~300-500ms
   - Пользователь видит спиннер
   - Через 500ms событие получает реальный ID → Undo разблокирован

4. **UX сообщение**:
   - Title: "Подождите"
   - Description: "Дождитесь завершения создания событий"
   - Variant: warning (жёлтый цвет)

   - Событие удаляется

## 📊 Технические детали

### Почему проверяем `id.startsWith('ev_temp_')`
```typescript
// Временные ID генерируются в createEvent():
const tempId = `ev_temp_${Date.now()}_${Math.random().toString(36).slice(2)}`;

// Примеры:
// - ev_temp_1700000000000_abc123
// - ev_temp_1700000001234_xyz789

// Реальные ID из БД:
// - ev123
// - ev456
```

### Почему этого достаточно
1. ✅ **Все pending события** имеют временный ID до завершения `createEvent`
2. ✅ **После завершения** событие получает реальный ID из БД
3. ✅ **Проверка** `events.some(e => e.id.startsWith('ev_temp_'))` покрывает ВСЕ pending операции:
   - Создание из двойного клика
   - Вставка (Paste)
   - Перетаскивание новых событий
   - Любые другие способы создания

### Альтернативные подходы (не использовались)

#### ❌ Проверять `pendingChanges.current.has(id)`
```typescript
// Проблема: pendingChanges отслеживает debounced save, а не CREATE операции
const hasPendingEvents = events.some(e => pendingChanges.current.has(e.id));
// Это НЕ даст нужную проверку для CREATE
```

#### ❌ Добавлять флаг `isPending` в событие
```typescript
// Проблема: Нужно отслеживать и обновлять флаг в нескольких местах
const event = { id: 'ev_temp_123', isPending: true, ... };
// После создания:
setEvents(prev => prev.map(e => e.id === tempId ? { ...e, isPending: false } : e));
// Сложнее и подвержено ошибкам
```
