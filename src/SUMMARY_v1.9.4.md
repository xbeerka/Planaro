## 🔧 Технические детали

### Архитектура

```
┌─────────────────────────────────────────────────┐
│         SchedulerContext.tsx                    │
├─────────────────────────────────────────────────┤
│                                                 │
│  Refs (защита от дублирования):                │
│  - lastLocalChangeRef (события)                │
│  - lastResourcesChangeRef (сотрудники)         │
│  - lastDepartmentsChangeRef (департаменты)     │
│  - lastProjectsChangeRef (проекты)             │
│                                                 │
│  Polling (useEffect):                          │
│  ┌──────────────────┐  ┌──────────────────┐   │
│  │ Events (10 sec)  │  │ Resources (15s)  │   │
│  └──────────────────┘  └──────────────────┘   │
│  ┌──────────────────┐  ┌──────────────────┐   │
│  │ Departments (15s)│  │ Projects (15s)   │   │
│  └──────────────────┘  └──────────────────┘   │
│                                                 │
│  CRUD операции:                                │
│  - create → устанавливают ref                  │
│  - update → устанавливают ref                  │
│  - delete → устанавливают ref                  │
│                                                 │
└─────────────────────────────────────────────────┘
         ↓                    ↓
    HTTP API           IndexedDB Cache
```

### Логика работы

1. **Polling запускается** при монтировании компонента
2. **Каждые N секунд**:
   - Проверяет: было ли локальное изменение < 2 сек назад?
   - Если да → пропускает polling
   - Если нет → запрашивает данные с сервера
3. **Сравнивает** через `JSON.stringify(prev) !== JSON.stringify(data)`
4. **Обновляет UI** только если данные изменились
5. **Обновляет кэш** в IndexedDB автоматически
6. **Останавливается** при размонтировании компонента

### Защита от дублирования

```typescript
// В CRUD операции
lastMyEntityChangeRef.current = Date.now();

// В polling
const timeSinceLastChange = Date.now() - lastMyEntityChangeRef.current;
if (timeSinceLastChange < 2000) {
  console.log('⏭️ Пропуск polling - недавно было локальное изменение');
  return;
}
```

## 🎓 Уроки и выводы

### Что сработало хорошо

1. **Simple Short Polling** оказался лучше Realtime Broadcast
   - Не требует дополнительных библиотек
   - Работает в любом окружении
   - Простая реализация и отладка

2. **Защита от дублирования** через refs
   - Предотвращает лишние запросы
   - Не ломает UX

3. **Разные интервалы** для разных типов данных
   - События (10 сек) - меняются чаще
   - Остальное (15 сек) - меняется реже
   - Оптимальный баланс производительность/актуальность

### Что можно улучшить в будущем

1. **Adaptive polling** - динамические интервалы
   - Чаще если пользователь активен
   - Реже если неактивен > 1 минуты

2. **Selective polling** - обновлять только видимые данные
   - Не polling скрытых отделов
   - Не polling если вкладка неактивна

3. **WebSocket fallback** - когда Figma Make будет поддерживать
   - Мгновенные обновления
   - Меньше нагрузки на сервер
