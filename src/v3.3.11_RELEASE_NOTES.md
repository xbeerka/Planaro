# 🎉 Release Notes v3.3.11

## 📦 Что нового

### 🛡️ Race Condition Fix - Защита Undo/Redo от одновременных операций

При быстром нажатии Ctrl+Z несколько раз подряд возникала race condition, когда второй Undo запускался ДО завершения первого. Это приводило к конфликтам синхронизации с сервером.

**v3.3.11 решает эту проблему через блокировку одновременных операций.**

---

## 🔧 Технические изменения

### 1. Новый Ref для блокировки
```typescript
// /components/scheduler/SchedulerMain.tsx
const isUndoRedoInProgressRef = useRef<boolean>(false);
```

### 2. Проверка в начале функций Undo/Redo
```typescript
const handleUndo = useCallback(async () => {
  // ✅ v3.3.11: Блокируем одновременные Undo операции
  if (isUndoRedoInProgressRef.current) {
    console.warn('⏸️ UNDO/REDO: Undo уже выполняется, ожидайте завершения...');
    return; // ← Блокируем повторный вызов
  }
  
  // ... rest of the code
}, [...]);
```

### 3. Finally блок для гарантированного снятия блокировки
```typescript
try {
  // ... вся логика undo/redo ...
} finally {
  // ✅ v3.3.11: Гарантированно снимаем блокировку даже при ошибках
  isUndoRedoInProgressRef.current = false;
  console.log('🔄 UNDO/REDO: 🔓 Блокировка снята');
}
```

## ✅ Что исправлено

### До v3.3.11 (БАГ):
```
Пользователь быстро нажимает Ctrl+Z 5 раз
   ↓
Все 5 Undo запускаются ОДНОВРЕМЕННО
   ↓
Конфликты синхронизации с сервером
   ↓
Ошибки, дублирование запросов, некорректные состояния
```

### После v3.3.11 (ИСПРАВЛЕНО):
```
Пользователь быстро нажимает Ctrl+Z 5 раз
   ↓
Первый Undo выполняется полностью
   ↓
Следующие 4 Undo блокируются (⏸️ сообщение в консоли)
   ↓
После завершения первого - следующий запускается
   ↓
События восстанавливаются ПОСЛЕДОВАТЕЛЬНО, без конфликтов
```

## 🎯 Преимущества

### 1. Защита от race conditions
- ✅ Невозможно запустить второй Undo/Redo пока первый не завершится
- ✅ Предотвращает конфликты при синхронизации с сервером
- ✅ Защита от дублирования запросов

### 2. Graceful degradation
- ✅ `finally` блок гарантирует снятие блокировки даже при ошибках
- ✅ Логирование: пользователь видит предупреждение в консоли
- ✅ Система не "зависает" даже если что-то пошло не так

### 3. Минимальные изменения
- ✅ Не меняет существующую логику Undo/Redo
- ✅ Добавлен только один `useRef` и проверка в начале функций
- ✅ Совместимо со всеми существующими исправлениями (v3.3.1-v3.3.10)

## 🎓 Важные моменты для разработчиков

### 1. КРИТИЧНО: finally блок ОБЯЗАТЕЛЕН
```typescript
try {
  // ... логика ...
} finally {
  // ✅ КРИТИЧНО: ВСЕГДА снимаем блокировку, даже при throw
  isUndoRedoInProgressRef.current = false;
}
```

**Почему**: Если не использовать `finally`, при ошибке блокировка не снимется и система "зависнет".

### 2. КРИТИЧНО: useRef вместо useState
```typescript
// ✅ ПРАВИЛЬНО - useRef (без ре-рендеров)
const isUndoRedoInProgressRef = useRef<boolean>(false);

// ❌ НЕПРАВИЛЬНО - useState вызовет лишние ре-рендеры
const [isUndoRedoInProgress, setIsUndoRedoInProgress] = useState(false);
```

**Почему**: Нам нужен только флаг для синхронизации, ре-рендеры не нужны.

### 3. КРИТИЧНО: Проверка В НАЧАЛЕ функции
```typescript
// ✅ ПРАВИЛЬНО - проверка ДО historyUndo()
if (isUndoRedoInProgressRef.current) return;
const state = historyUndo();

// ❌ НЕПРАВИЛЬНО - historyUndo() уже изменит стек истории
const state = historyUndo();
if (isUndoRedoInProgressRef.current) return;
```

**Почему**: Если проверка после `historyUndo()`, стек истории уже изменится даже при блокировке.
